Issue 127 日本語解説資料

SNS プロダクト境界と運用方針
日本語解説書

この文書は、Issue 127 で整理された「簡易 SNS の初回リリースで何を決め、何をまだ決めないか」を、実装担当者とレビュー担当者が素早く共有できるように説明用へ再構成した資料である。元の Issue 本文にある作業契約や実務向けチェックリストは参照元として残しつつ、本書では判断の意味と後続課題への影響を読みやすくまとめる。

対象 SNS 初回リリースの計画と実装準備を担当する人
主題 公開閲覧と認証済み投稿を中心にした狭い初期境界
目的 認証、API、永続化、モデレーションの後続判断を同じ前提にそろえる

1. この Issue が必要になった理由

Issue 127 は、SNS を実装する前に製品境界を先に固定するための計画用課題である。

現行の portal は、静的配信優先、公開優先、エンドユーザー認証なし、アプリケーション永続化なしを前提とした設計で閉じている。一方、簡易 SNS は「投稿できるか」「誰に見えるか」「削除や非表示をどう扱うか」という判断が先に固定されていないと、認証、API、データ保存、運用ポリシーがそれぞれ別の仮定で進んでしまう。

そのため Issue 127 の役割は、SNS 実装そのものではなく、初回リリースの製品境界を先に固定することにある。ここで扱うのは、投稿者の役割モデル、公開範囲、削除方針、最小限のモデレーション線、対象外項目の 5 点であり、認証基盤やデータベース製品の選定はまだ先の話として切り分ける。

読むときの前提

この文書は「何を実装するか」よりも「どこまでを初回リリースに含めるか」を決める資料である。実装詳細の決定資料ではないため、ベンダーやクラウド固有の命名、トークン形式、データベース種別までは踏み込まない。

2. 初回リリースの最終判断

この Issue で固定した判断は、後続 Issue が前提を再定義しないための基準線になる。

投稿者 投稿できるのは認証済みメンバーのみとし、閲覧者は閲覧専用にとどめる。
公開範囲 読み取りは公開タイムラインを基本とし、限定公開の閲覧分岐は初回リリースに含めない。
削除方針 物理削除を標準にせず、非表示または論理削除相当の扱いを優先する。
最小限のモデレーション線 スパム、攻撃的な文面、明らかな不正入力、過剰投稿への最小介入線を設ける。

3. 役割ごとの境界

閲覧者は読むだけ、メンバーは投稿できる、運用担当者は最低限のモデレーションを担う。この 3 層で初回リリースの責務差分を固定する。
役割 できること できないこと 意味
閲覧者 公開タイムラインの閲覧 投稿、削除、モデレーション対象の操作 初回リリースの価値は公開閲覧に置き、導線の摩擦を増やさない
メンバー 認証後の投稿 運用担当者権限を要する非表示、全体削除 投稿経路だけを保護対象の操作面として扱う
運用担当者 非表示、論理削除相当、最小限のモデレーション操作 本格的な審査体制、重い異議申立てフロー 初回リリースでは最小限の安全運用線に限定する

4. 公開閲覧と認証済み投稿の考え方

この Issue の核は、読む行為と書く行為を明確に切り分けることにある。公開タイムラインはログインなしで読める状態を維持しつつ、投稿は認証済みメンバーだけに許可する。これにより、初回リリースの導線はわかりやすく保たれ、SNS として最低限必要な不正利用対策も投稿経路側に集中させやすくなる。

今回の対象 公開タイムラインの閲覧、認証済みメンバーの投稿、運用担当者による非表示または論理削除相当
今回の対象外 限定公開の閲覧分岐、クローズドグループ、ダイレクトメッセージ、フォロー関係、ランキング

5. 削除方針と最小限のモデレーション線

初回リリースの削除方針は、利用者の操作自由度を広げることよりも、運用時に事故を起こしにくいことを優先する。そのため物理削除を通常系にせず、非表示や論理削除相当の扱いを基準に置く。これにより、不適切投稿の非表示、監査のための状態保持、復旧判断の余地を残しやすくなる。

論点 初回リリースの判断 理由
投稿者本人の編集 対象外 履歴管理と競合解決を増やすため
投稿者本人の削除 優先度を下げ、運用担当者管理を先行候補とする 誤消去と不正利用対応の両面で初期運用を単純化するため
スパム / 攻撃的な文面 最小限のモデレーション線に含める 公開タイムラインの品質維持に不可欠なため
重い異議申立てフロー 対象外 運用フローが過大になり初回リリースの境界を超えるため
注意点

最小限のモデレーション線を薄くしすぎると公開閲覧の価値が下がり、逆に厚くしすぎると初回リリースの実装範囲が膨らむ。Issue 127 はこの中間点として「最低限の安全運用線」だけを固定している。

6. 初回リリースに含めないもの

対象外項目を明文化する目的は、単に機能を削ることではない。後続 Issue が範囲肥大を起こさず、認証、API、永続化、運用設計を狭い範囲に保つためである。

相互作用機能 いいね、リアクション、返信スレッドの深掘り、フォロー関係
非公開機能 ダイレクトメッセージ、限定公開の閲覧範囲、複雑な権限分岐
コンテンツ拡張 メディア投稿、編集履歴、拡張プロフィール連携
高度な配信 推薦表示、ランキング、リアルタイム通知、マルチクラウドの同時書き込み経路

7. この判断が後続課題に与える影響

Issue 127 で境界を固定すると、後続 Issue は「SNS に何が必要か」を毎回再定義せずに済む。特に認証境界の再整理、メッセージのドメインモデル、バックエンドと API の基準線に関する Issue は、この判断を正本として参照する前提で進められる。

後続テーマ Issue 127 から引き継ぐ前提 まだ決めないこと
認証 公開閲覧、認証済み投稿、運用担当者操作の境界 認証基盤の製品選定、トークン詳細
永続化 メッセージ公開範囲、削除の意味づけ、モデレーション対象の状態 データベース製品、索引実装の詳細
API 設計 投稿、一覧取得、非表示や削除の最小操作面 スキーマの版管理詳細、基盤固有の対応付け

8. 実装チーム向けの読み方

  1. この文書は「機能一覧」ではなく「実装境界」の文書として読む。
  2. 初回リリースに必要な最小線だけを拾い、魅力的でも対象外の機能は別 Issue に逃がす。
  3. 公開閲覧を守ることと認証済み投稿の安全性を両立させることを優先する。
  4. 運用担当者の行為は便利機能ではなく、最小限のモデレーション線として扱う。
  5. 後続 Issue で前提を変えるなら、Issue 127 を暗黙に上書きせず、別の確定判断として切り出す。
