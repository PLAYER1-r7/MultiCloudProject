# 手動デプロイ判断基準

## この文書を使う場面

- CI/CD が詰まっているとき、リリースが承認待ちのとき、または実行時チューニング変更が実装変更ではなく運用調整に見えるときに使う
- 手動デプロイや実行時パラメータ上書きの前に使う

## 判断手順

1. なぜ自動化で足りないかを確認する
2. ロールバックと承認の準備を確認する
3. 問題が設計変更ではなく運用判断かを確認する
4. 実行前に手動手順または実行時上書きを記録する

## 原則

デフォルトは CI/CD 自動化を使います。

## 手動デプロイが許可される条件

- CI が詰まっている、または緊急復旧が必要
- 変更範囲が限定されている
- ロールバックが準備されている
- 必要な承認と認証がある

緊急復旧で手動デプロイを許可できるのは、障害の発生源が設定、デプロイアーティファクト、環境変数など特定のレイヤーに絞り込めている場合に限る。発生源を特定レイヤーに絞り込めない場合は「根本原因不明」として扱い、ハードストップルールを適用する。

## 手動デプロイ禁止条件

- 根本原因が不明
- 本番影響が未管理
- ロールバックが不明
- 境界違反がある

## アプリ固有ランタイム判断ルール

実装設計そのものの変更ではなく、ランタイム挙動に効くアプリ固有の運用調整判断はこの文書に置きます。

### exam-solver の Timeout 基準

- 現在の参照基準は、バックエンド `120s`、フロントエンド `120s`
- この基準は、OCR と多段推論で合計 `60-90s` 程度かかり得る前提に基づく
- timeout は `120s` -> `150s` -> `180s` の順で段階的に引き上げる
- timeout を上げるのは、原因が timeout であり、ロジック・認証・インフラ障害ではないと確認した後に限る

### 判断ゲート

- timeout 値を変える前に、実測 latency とエラーログを優先して確認する
- timeout を変更する場合は、バックエンドとフロントエンドの期待値を揃える
- `180s` でも不足する場合は、timeout 問題として扱い続けず、最適化または設計課題としてエスカレーションする

## 実行記録

`08_ESCALATION_AND_HANDOFF.md` で定義された正規 Execution Record 形式を使って記入します。

`Next action` で reviewer / approval owner の境界を書くときは `ROLE_HANDOFF_OWNERSHIP.md` に合わせます。

## 記入例

```text
Document: 20_MANUAL_DEPLOY_DECISION_CRITERIA
Scope: Issue #487 に対する sns frontend_react staging hotfix の手動デプロイ判断
Outcome: 手動デプロイ承認
Actions taken: sns 向け CI 経路が使えないこと、ロールバックがあること、変更が frontend_react 資産に限定されることを確認し、deploy-sns-aws.yml 代替手順を選んだ
Evidence: deploy-sns-aws.yml が runner 障害で停止; frontend_react 成果物ビルドは Issue #487 向けに成功した
Risks or blockers: CI 復旧までは sns の手動経路を staging 限定に保つ必要がある
Next action: deploy-sns-aws.yml の代わりに文書化された sns 向け staging 専用手動デプロイ手順を実行し、sns-reviewer 共有後に sns-approval-owner のサインオフを得る
```
